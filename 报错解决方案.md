## [2026-03-22] RocketMQTemplate Bean 找不到导致启动失败

### 报错信息

```
APPLICATION FAILED TO START

Description:
Parameter 0 of constructor in com.fireworks.service.cart.CartMqProducer
required a bean of type 'org.apache.rocketmq.spring.core.RocketMQTemplate'
that could not be found.

Action:
Consider defining a bean of type
'org.apache.rocketmq.spring.core.RocketMQTemplate' in your configuration.
```

后续演变为：

```
Parameter 1 of constructor in com.fireworks.service.impl.OmsCartServiceImpl
required a bean of type 'com.fireworks.service.cart.CartMqProducer'
that could not be found.
```

### 影响模块

- `fireworks-admin`（8081 端口）

---

### 根本原因分析

#### 第一层：`RocketMQTemplate` Bean 为何不创建

`RocketMQAutoConfiguration` 源码中，`rocketMQTemplate` Bean 的注册条件为：

```java
@Bean(destroyMethod = "destroy")
@Conditional(ProducerOrConsumerPropertyCondition.class)
@ConditionalOnMissingBean(name = ROCKETMQ_TEMPLATE_DEFAULT_GLOBAL_NAME)
public RocketMQTemplate rocketMQTemplate(RocketMQMessageConverter rocketMQMessageConverter) {
    RocketMQTemplate rocketMQTemplate = new RocketMQTemplate();
    if (applicationContext.containsBean(PRODUCER_BEAN_NAME)) {
        rocketMQTemplate.setProducer(...);
    }
    if (applicationContext.containsBean(CONSUMER_BEAN_NAME)) {
        rocketMQTemplate.setConsumer(...);
    }
    ...
    return rocketMQTemplate;
}
```

其中 `ProducerOrConsumerPropertyCondition` 是一个 `AnyNestedCondition`：

```java
static class ProducerOrConsumerPropertyCondition extends AnyNestedCondition {

    public ProducerOrConsumerPropertyCondition() {
        super(ConfigurationPhase.REGISTER_BEAN);
    }

    @ConditionalOnBean(DefaultMQProducer.class)
    static class DefaultMQProducerExistsCondition {}

    @ConditionalOnBean(DefaultLitePullConsumer.class)
    static class DefaultLitePullConsumerExistsCondition {}
}
```

`AnyNestedCondition` 语义：**两个内部条件至少满足一个，外层条件才成立**。
即：容器中存在 `DefaultMQProducer` Bean **或** `DefaultLitePullConsumer` Bean，`RocketMQTemplate` 才会被创建。

---

#### 第二层：`DefaultMQProducer` Bean 为何不创建

```java
@Bean(PRODUCER_BEAN_NAME)
@ConditionalOnMissingBean(DefaultMQProducer.class)
@ConditionalOnProperty(prefix = "rocketmq", value = {"name-server", "producer.group"})
public DefaultMQProducer defaultMQProducer(RocketMQProperties rocketMQProperties) { ... }
```

`@ConditionalOnProperty` 同时要求 **`rocketmq.name-server`** 和 **`rocketmq.producer.group`** 都存在。

Admin 模块最初的配置只有：

```yaml
rocketmq:
  name-server: 127.0.0.1:9876
```

缺少 `rocketmq.producer.group`，条件不满足，`DefaultMQProducer` Bean 不创建。

同理，`DefaultLitePullConsumer` 需要 `pull-consumer.group` 和 `pull-consumer.topic`，也未配置，同样不创建。

---

#### 完整失败链路

```
admin application.yml 缺少 rocketmq.producer.group
  → @ConditionalOnProperty 不满足
  → DefaultMQProducer Bean 不创建
  → ProducerOrConsumerPropertyCondition 两个内部条件均为 false
  → RocketMQTemplate Bean 不创建（方法体根本不会执行）
  → CartMqProducer 构造函数注入 RocketMQTemplate 失败
  → OmsCartServiceImpl 注入 CartMqProducer 失败
  → 启动报错
```

> **注意**：`rocketMQTemplate` 方法体里虽然用的是 `applicationContext.containsBean()` 做运行时判断，
> 看起来与 `producer.group` 无关——但这是迷惑点。
> Spring 在决定是否注册这个 Bean 时，先评估 `@Conditional`，结果为 false 则直接跳过，
> 方法体**永远不会被执行**，`RocketMQTemplate` Bean 根本不会出现在容器中。

---

#### 关于 Gradle 依赖传递（已验证，非问题原因）

曾怀疑 Gradle `implementation` 不传递导致 admin 拿不到 `rocketmq-spring-boot-starter`，**该判断有误**。

实际验证（`gradle :fireworks-admin:dependencies --configuration runtimeClasspath`）证明：
`project(':fireworks-service')` 的运行时依赖（含 `rocketmq-spring-boot-starter`）**会传递到** admin 的 `runtimeClasspath`。
Gradle `implementation` 只隔离编译期符号可见性，不影响运行时 classpath 的继承。

因此 admin 的 `build.gradle` **不需要**显式声明 `rocketmq-spring-boot-starter`。

#### 关于 `@ConditionalOnBean` 误用（弯路，勿重蹈）

曾尝试给 `CartMqProducer` 加 `@ConditionalOnBean(RocketMQTemplate.class)`，
期望在没有 `RocketMQTemplate` 时跳过注册。但 `CartMqProducer` 被 `OmsCartServiceImpl` 强依赖，
条件不满足时 `CartMqProducer` 消失，`OmsCartServiceImpl` 又报找不到它，陷入新的错误循环。

> **结论**：`@ConditionalOnBean` 不能用在被其他必须注册的 Bean 依赖的组件上。

---

### 解决步骤

在 `fireworks-admin/src/main/resources/application.yml` 补上 `producer.group`：

```yaml
rocketmq:
  name-server: 127.0.0.1:9876
  producer:
    group: cart-producer-group-admin   # 与 api 的 producer group 区分，避免冲突
    send-message-timeout: 3000
```

补上 `producer.group` 后：

```
DefaultMQProducer Bean 创建成功
  → ProducerOrConsumerPropertyCondition = true
  → RocketMQTemplate Bean 创建成功
  → CartMqProducer 注入成功
  → 启动正常
```

> Admin 模块本身不发送购物车 MQ 消息，但 `scanBasePackages` 扫描了 `com.fireworks.service`，
> `CartMqProducer` 会被注册，因此必须提供完整的 RocketMQ Producer 配置使 `RocketMQTemplate` Bean 得以创建。
> 使用独立的 `group` 名（`cart-producer-group-admin`）与 api 模块的 `cart-producer-group` 区分。

---

### 附：Consumer Group 重复隐患说明

Admin 和 api 都扫描了 `com.fireworks.service` 包，`CartPersistConsumer`
（`consumerGroup = "cart-consumer-group"`）会在两个模块中各自注册一个 Consumer。

若两个模块**同时运行**，同一条购物车 MQ 消息可能被两个模块各消费一次，导致 MySQL **重复落库**。

**规避方式**：开发阶段不同时启动 admin 和 api；生产环境应将购物车 Consumer 拆到独立的 worker 进程。

### 核心教训

| 知识点 | 说明 |
|---|---|
| `RocketMQTemplate` 创建条件 | 由 `ProducerOrConsumerPropertyCondition`（`AnyNestedCondition`）控制，需容器中存在 `DefaultMQProducer` 或 `DefaultLitePullConsumer` Bean，而非直接依赖配置项 |
| `DefaultMQProducer` 创建条件 | `@ConditionalOnProperty` 要求 `rocketmq.name-server` 和 `rocketmq.producer.group` 同时存在，缺一不可 |
| 方法体与 `@Conditional` 的执行顺序 | `@Conditional` 在 Bean 注册阶段评估，为 false 时方法体永远不执行，不要被方法体内的逻辑误导 |
| Gradle `implementation` 与运行时传递 | 只隔离编译期符号，运行时 classpath 仍向上传递，无需在 admin/api 重复声明 service 的依赖 |
| `@ConditionalOnBean` 使用限制 | 不能用在被其他必须注册的 Bean 依赖的组件上，否则引发新的 Bean 缺失错误 |
| Consumer Group 重复 | 多进程同一 group 争抢消息，可能导致重复消费，需谨慎 |
