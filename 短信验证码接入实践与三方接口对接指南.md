# 短信验证码接入实践与三方接口对接指南

> 本文档记录「烟火邻里」项目接入阿里云短信验证码的完整过程，以及对接第三方 API 的正确思路与向 AI 提问的最佳实践。

---

## 一、功能实现概览

### 1.1 已实现功能

| 模块 | 说明 |
|------|------|
| **Lua 脚本限流** | `scripts/sms-rate-limit.lua`，Redis 原子执行：60 秒间隔 + 每日 10 次上限 |
| **SmsService** | 发送验证码、校验限流、存 Redis（`auth:code:phone:{phone}`，5 分钟有效） |
| **阿里云接入** | 使用 **dypnsapi**（号码认证服务）`SendSmsVerifyCode` 接口，非 dysmsapi |
| **Controller** | `POST /api/sms/sendVerifyCode?phone=xxx`，无需 Token |
| **统一异常处理** | `GlobalExceptionHandler` 处理 `SmsLimitException`、`SmsSendException` 等 |

### 1.2 关键配置

```yaml
# application.yml
aliyun:
  sms:
    access-key-id: xxx
    access-key-secret: xxx
    sign-name: 速通互联验证码    # 需在阿里云控制台申请
    template-code: "100001"      # dypnsapi 使用数字格式
```

### 1.3 依赖

```groovy
// fireworks-common / fireworks-service
implementation 'com.aliyun:dypnsapi20170525:2.0.0'
```

## 二、踩坑与纠正过程

### 2.1 错误一：选错 SDK 产品

| 产品 | 用途 | 模板格式 | 接口 |
|------|------|----------|------|
| **dysmsapi**（短信服务） | 通用短信发送 | `SMS_123456789` | `SendSms` |
| **dypnsapi**（号码认证服务） | 验证码、一键登录 | `100001` | `SendSmsVerifyCode` |

**问题**： initially 使用了 dysmsapi，出现 `isv.SMS_TEMPLATE_ILLEGAL`、`isv.SMS_SIGNATURE_ILLEGAL` 等错误，签名和模板在控制台申请的格式与 dysmsapi 要求不一致。

**解决**：参照阿里云官方下载的 SDK 示例，改用 **dypnsapi** 的 `SendSmsVerifyCode` 接口。

### 2.2 错误二：包名与 Maven 坐标混淆

- `alibabacloud-dypnsapi20170525`：异步 SDK，包名可能不同
- `dypnsapi20170525`：同步 SDK，包名 `com.aliyun.dypnsapi20170525`（与官方示例一致）

**解决**：使用 `com.aliyun:dypnsapi20170525:2.0.0`，与官方 Sample 保持一致。

### 2.3 关键转折：提供官方 SDK 示例

耗时近两小时后，提供从阿里云官网下载的完整 SDK 示例项目路径：

```
@c:\Users\Administrator\Desktop\0db7ada6-1f77-42d2-a150-eb7c29ea5efc-Java
参照该发送短信的项目帮我修改代码，这是我从官网下载的完整的SDK示例
```

**对照官方示例后**，代码得以快速修正：Client 类型、请求类、接口调用方式全部对齐，问题迎刃而解。

## 三、对接第三方 API 的正确思路

### 3.1 总体流程

```
1. 明确业务需求（如：发送短信验证码）
       ↓
2. 在官网确认对应产品与 API（dysmsapi vs dypnsapi）
       ↓
3. 下载官方 SDK 示例（Java / 对应语言）
       ↓
4. 本地运行官方示例，确认环境与配置正确
       ↓
5. 将示例中的 Client、Request、调用方式迁移到项目
       ↓
6. 封装业务逻辑（限流、存储、异常处理）
```

### 3.2 关键检查清单

| 检查项 | 说明 |
|--------|------|
| **产品线** | 同一云厂商下可能有多条产品线（如 dysmsapi / dypnsapi），选对再开发 |
| **Maven/Gradle 坐标** | 使用官方文档或 SDK 示例中的依赖，避免搜到过期/错误版本 |
| **包名与类名** | 不同 artifact 可能包名不同，以实际能 import 为准 |
| **配置格式** | 签名、模板、参数格式以控制台实际为准，不要凭记忆猜测 |
| **官方示例优先** | 示例代码代表当前推荐用法，优先参考 |

### 3.3 避免的常见误区

1. **不要仅靠搜索引擎**：搜到的 blog / 问答可能是旧版或错误产品线
2. **不要跳过官方示例**：先跑通示例，再往项目里迁，效率更高
3. **不要混淆相似产品**：如 dysmsapi 与 dypnsapi 名称相近但用途不同
4. **不要硬编码凭证**：使用环境变量或配置中心，避免泄露

## 四、向 AI 提问的最佳实践

### 4.1 推荐提问方式

**❌ 低效示例：**

> 帮我接阿里云短信

问题：产品线不明、接口不明、无参考代码，AI 容易猜错或给出过时方案。

**✅ 高效示例：**

> 参照阿里云官方 SDK 示例 @路径/xxx-Java 帮我接入短信验证码。
> 官方示例使用 dypnsapi 的 SendSmsVerifyCode，请按相同方式修改我项目中的 SmsServiceImpl 和配置。

优势：提供官方示例、明确接口与产品线，便于 AI 对齐实现。

### 4.2 提问模板（可复用）

```
【背景】项目需对接 XXX 的 YYY 功能（如：阿里云发送短信验证码）

【已有】我从官网下载了官方 SDK 示例：@路径/项目名

【约束】JDK 1.8 / Spring Boot 2.7 / Gradle

【需求】请参照该示例修改我项目中的 ZZZ 类，使用相同的 Client、Request 和调用方式
```

### 4.3 提供官方示例的价值

| 维度 | 价值 |
|------|------|
| **权威性** | 代表当前官方推荐用法 |
| **可对照** | AI 可直接对比并迁移，减少猜测 |
| **省时** | 避免多次试错，显著缩短对接时间 |
| **版本一致** | Maven 坐标、包名、接口用法与示例一致 |

## 六、总结

1. **对接第三方 API**：优先使用官方示例，选对产品线与 SDK，再迁入项目。
2. **向 AI 求助时**：提供官方示例路径、明确产品和接口，可大幅提升效率和准确率。
3. **本次实践**：提供 `0db7ada6-1f77-42d2-a150-eb7c29ea5efc-Java` 示例后，问题得以迅速解决，是本次接入的关键转折点。
