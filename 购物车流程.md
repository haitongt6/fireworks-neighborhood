### 4.1.1 OmsCartServiceImpl#add 并发安全详细流程（Redisson 锁版）

```
POST /api/cart/add { productId, quantity }
        |
        v
 OmsCartServiceImpl.add(userId, param)
        |
        v
 lockKey = "cart:lock:" + userId + ":" + productId
 RLock lock = redissonClient.getLock(lockKey)
        |
        v
 lock.tryLock(等待3s, 持锁5s)
        |
   +-----------+------------+
   |                        |
 拿到锁                  未拿到锁（3s内)
   |                        |
   v                        v
[商品校验]          throw IllegalStateException
 · selectById(productId)    "操作频繁，请稍后重试"
 · product == null -> 商品不存在
 · status != 1   -> 商品已下架
   |
   v
[读 Redis 当前数量]（锁保护内，串行安全）
 CartRedisHelper.getItem(userId, productId)
 currentQty = existing != null ? existing.quantity : 0
 newQty = currentQty + addQty
   |
   v
[库存校验]
 newQty > product.stock -> 库存不足
   |
   v
[限购校验]
 newQty > limitPerUser  -> 超出限购数量
   |
   v
 extractFirstImage(product.images) -> imageSnapshot
   |
   v
 CartItemEntry(newQty, price, title, imageSnapshot)
   |
   +------------------+
   |                  |
   v                  v
Redis               @Async
setItem()          asyncUpsert()
Hash PUT           MySQL upsert
TTL 刷新           重试 3 次
   |
   v
 finally 块
 locked && lock.isHeldByCurrentThread()
        -> lock.unlock()
   |
   v
 return（Controller 层包装 Result.success）
```

**并发安全关键点说明：**

| 要点 | 说明 |
|---|---|
| 锁粒度 | `userId + productId` 双维度，不同商品互不阻塞 |
| tryLock 超时 | 等待最多 3 秒，防止线程无限堆积 |
| 持锁上限 | 最多持锁 5 秒，防止业务异常导致死锁 |
| isHeldByCurrentThread | finally 中判断后再 unlock，防止解锁他人的锁 |
| InterruptedException | 捕获后恢复中断状态，不吞异常 |
| 串行保证 | 同一用户同一商品的并发请求在锁内排队执行，数量累加结果精确 |

## 四点五、RocketMQ 顺序落库详解

### 背景与问题

原方案使用 `@Async` + 线程池异步落库 MySQL，存在以下问题：
- 落库失败仅打日志，消息丢失无兜底
- 多线程并发执行，同一用户的 add/update/remove 操作落库顺序无法保证
- 无死信队列，无法感知和处理持续失败的消息

### 解决方案：RocketMQ 顺序消息

**核心原理：** 同一用户的所有写操作消息，通过 `hashKey = userId` 路由到同一个 MessageQueue，Consumer 以 `ORDERLY`（顺序）模式串行消费，保证落库顺序与 Redis 操作顺序完全一致。

### RocketMQ 安装配置（本地开发）

**版本要求：** RocketMQ 4.9.x（兼容 Java 8），下载地址：
```
https://mirrors.aliyun.com/apache/rocketmq/4.9.8/rocketmq-all-4.9.8-bin-release.zip
```

**解压路径：** `D:\dev-tools\rocketmq-4.9.8`

**环境变量：**
```
ROCKETMQ_HOME = D:\dev-tools\rocketmq-4.9.8
NAMESRV_ADDR  = 127.0.0.1:9876
```

**调整 JVM 内存**（本地开发避免 OOM）：
- `bin\runserver.cmd`：`-Xms256m -Xmx256m -Xmn128m`
- `bin\runbroker.cmd`：`-Xms256m -Xmx256m -Xmn128m`

**启动顺序：**
```bash
# 1. 先启动 NameServer（保持窗口不关闭）
D:\dev-tools\rocketmq-4.9.8\bin\mqnamesrv.cmd
# 看到：The Name Server boot success

# 2. 再启动 Broker（保持窗口不关闭）
D:\dev-tools\rocketmq-4.9.8\bin\mqbroker.cmd -n 127.0.0.1:9876 autoCreateTopicEnable=true
# 看到：The broker boot success
```

**Spring Boot 配置**（`fireworks-api/src/main/resources/application.yml`）：
```yaml
rocketmq:
  name-server: 127.0.0.1:9876
  producer:
    group: cart-producer-group
    send-message-timeout: 3000
```

**Gradle 依赖**（`fireworks-service/build.gradle`）：
```groovy
implementation 'org.apache.rocketmq:rocketmq-spring-boot-starter:2.2.3'
```

### 新增文件说明

| 文件 | 职责 |
|---|---|
| `CartPersistMessage.java` | 消息体，包含操作类型（UPSERT/DELETE/CLEAR）、userId、productId 等字段 |
| `CartMqProducer.java` | 生产者，封装 `sendUpsert` / `sendDelete` / `sendClear` 三个方法，使用 `sendOneWayOrderly` 发送 |
| `CartPersistConsumer.java` | 消费者，`@RocketMQMessageListener(consumeMode=ORDERLY)` 串行落库，失败抛异常触发重试 |

### 顺序消息落库完整流程

```
写操作请求（add / update / remove / clear）
        |
        v
① OmsCartServiceImpl 写 Redis（同步，主流程）
        |
        v
② CartMqProducer.sendOneWayOrderly(topic, msg, hashKey=userId)
        |
        v
   RocketMQ Broker
   按 userId % queueSize 路由到固定 MessageQueue
        |
        v
③ CartPersistConsumer.onMessage()（ORDERLY 串行消费）
        |
   +----+----+
   |         |         |
   v         v         v
UPSERT    DELETE    CLEAR
查已有     按userId  按userId
存在->     +productId 删全部
updateById 删单条
不存在->
insert
   |
   v
消费成功 -> ACK
   |
   v
消费失败（抛异常）
   |
   v
RocketMQ 自动重试（顺序消费阻塞当前队列直到成功）
   |
超过最大重试次数
   |
   v
进入死信队列（%DLQ%cart-consumer-group）
可通过 RocketMQ Dashboard 查看并人工处理
```

### 与原 @Async 方案对比

| 对比维度 | 原 @Async 方案 | RocketMQ 顺序消息方案 |
|---|---|---|
| 落库失败处理 | 最多重试 3 次，失败仅打日志，消息丢失 | 无限重试，超限进死信队列，消息不丢 |
| 操作顺序保证 | 无保证，线程池并发执行可能乱序 | 同一用户消息严格串行，顺序与 Redis 一致 |
| 可观测性 | 只有日志 | RocketMQ Dashboard 可查积压量、消费进度、死信 |
| 系统依赖 | 无额外依赖 | 需要 RocketMQ NameServer + Broker 运行 |

## 五、对标大厂的问题与差距

### 5.1 数据一致性问题

| # | 问题 | 当前实现 | 风险 | 大厂方案 | 状态 |
|---|---|---|---|---|---|
| 1 | Redis 与 MySQL 弱一致 | RocketMQ 顺序消息落库，失败自动重试，超限进死信队列 | 高 | MQ + 死信队列兜底 | ✅ 已完成 |
| 2 | 并发加购数量不准 | Redisson 分布式锁保护「读-校验-写」，锁粒度 userId+productId | 中 | Lua 脚本原子操作 / Redisson 锁 | ✅ 已完成 |
| 3 | 价格快照不刷新 | 加购时记录一次，后续不更新 | 中 | 结算时重新比价，变动提示用户 | ⏳ 待完成 |
| 4 | TTL 当天过期 | 每天 23:59:59 失效 | 低 | 7~30 天 TTL，活跃用户自动续期 | ⏳ 待完成 |

### 5.2 业务功能缺失

| # | 缺失 | 影响 | 参考方案 |
|---|---|---|---|
| 1 | 无 SKU 规格 | 无法区分商品不同规格 | 以 (productId, skuId) 为唯一键 |
| 2 | 无游客购物车 | 未登录无法加购 | localStorage 存储，登录后 merge |
| 3 | 选中状态不持久 | 刷新后丢失 | Redis 持久化或 sessionStorage |
| 4 | 无促销关联 | 看不到满减优惠 | 接口返回命中促销规则和优惠金额 |
| 5 | 无失效自动清理 | 下架商品堆积 | 定时任务扫描并通知用户 |
| 6 | 无实时库存推送 | 库存变化不及时感知 | WebSocket / SSE 推送 |
| 7 | 无跨设备同步 | 手机/PC 购物车不一致 | 服务端为唯一数据源 + WS |
| 8 | 无凑单推荐 | 结算无差价提示 | 根据金额推荐凑满减商品 |

### 5.3 性能问题

| # | 问题 | 当前实现 | 优化方向 |
|---|---|---|---|
| 1 | N+1 查询 | buildVOList 逐条查 pms_product | selectBatchIds() 批量查询 |
| 2 | 商品信息无缓存 | 每次实时查 MySQL | 热点商品信息缓存到 Redis |
| 3 | 库存校验走 DB | add 时同步查数据库 | 热点库存放 Redis，减少 DB 压力 |

## 六、后续优化路线图

### 阶段一：稳定性（高优先级）

- [ ] **N+1 修复**：`buildVOList` 改为 `selectBatchIds` 批量查商品
- [x] **异步落库可靠性**：引入 RocketMQ 顺序消息，Consumer 串行落库，失败自动重试，超限进死信队列

  ```lua
  local qty = redis.call('HGET', key, productId)
  local newQty = (qty or 0) + addQty
  -- 校验库存后

  ```
