# 烟火邻里 - 管理员登录流程文档

> 本文档详细描述后台管理系统（fireworks-admin）的登录流程，重点讲解 Spring Security 配置、过滤器链及 JWT 认证机制，便于后期快速理解与维护。

## 四、Spring Security 核心配置详解

### 4.1 SecurityConfig 类结构

路径：`fireworks-admin/src/main/java/com/fireworks/admin/config/SecurityConfig.java`

继承 `WebSecurityConfigurerAdapter`，是 Spring Security 的配置入口，主要完成：

| 配置项 | 作用 |
|--------|------|
| `@EnableWebSecurity` | 开启 Spring Security Web 支持 |
| `@EnableGlobalMethodSecurity(prePostEnabled = true)` | 开启方法级权限控制，支持 `@PreAuthorize` |
| `configure(AuthenticationManagerBuilder)` | 配置认证管理器：UserDetailsService + PasswordEncoder |
| `configure(HttpSecurity)` | 配置 HTTP 安全规则、异常处理器、过滤器 |

### 4.2 认证管理器配置

```java
auth.userDetailsService(adminUserDetailsService)   // 登录时加载用户
    .passwordEncoder(passwordEncoder());          // BCrypt 密码比对
```

- **AdminUserDetailsService**：实现 `UserDetailsService`，根据用户名从 DB 加载 `AdminUserDetails`（含权限、角色）
- **PasswordEncoder**：`BCryptPasswordEncoder`，用于密码加密存储与登录比对

> 注意：本项目登录**不走** Spring Security 默认的 `UsernamePasswordAuthenticationFilter`，而是自定义 Controller 调用 `UmsAdminService.login()`，因此 `AuthenticationManagerBuilder` 主要提供 `PasswordEncoder` 给 Service 层使用。

### 4.3 HTTP 安全规则

| 规则 | 含义 |
|------|------|
| `csrf().disable()` | 禁用 CSRF，前后端分离 REST 场景不需要 |
| `sessionCreationPolicy(STATELESS)` | 无 Session，完全依赖 JWT 无状态认证 |
| `antMatchers(POST, "/admin/login").permitAll()` | 登录接口放行，无需认证 |
| `antMatchers(GET, "/admin/publicKey").permitAll()` | 获取公钥接口放行 |
| `antMatchers(OPTIONS).permitAll()` | CORS 预检请求放行 |
| `anyRequest().authenticated()` | 其余所有接口必须已认证 |
| `formLogin().disable()` | 禁用默认表单登录页 |
| `httpBasic().disable()` | 禁用 HTTP Basic，改用 JWT Bearer |

### 4.4 异常处理器

| 处理器 | 触发场景 | 返回 |
|--------|----------|------|
| **RestAuthenticationEntryPoint** | 未认证（无 Token 或 Token 无效）访问受保护接口 | 401 JSON: `{"code":401,"message":"未登录或 Token 已过期，请重新登录"}` |
| **RestAccessDeniedHandler** | 见 4.5 节（URL 级鉴权失败） | 403 JSON: `{"code":403,"message":"权限不足，禁止访问"}` |
| **GlobalExceptionHandler** | 见 4.5 节（方法级 @PreAuthorize 失败） | 403 JSON: `{"code":403,"message":"权限不足，禁止访问"}` |

### 4.5 403 异常处理：RestAccessDeniedHandler 与 GlobalExceptionHandler 的区别

Spring Security 有两套鉴权机制，403 异常由**不同组件**在不同阶段处理：

| 鉴权类型 | 触发位置 | 异常抛出位置 | 实际处理者 |
|----------|----------|--------------|------------|
| **URL 级鉴权** | `FilterSecurityInterceptor`（过滤链中） | 过滤链 | **RestAccessDeniedHandler** |
| **方法级鉴权** | `@PreAuthorize`（Controller 方法调用前） | DispatcherServlet / MVC 层 | **GlobalExceptionHandler** 的 `handleAccessDeniedException` |

**当前项目情况：**

配置中仅有 `anyRequest().authenticated()`，未对具体 URL 配置 `hasAuthority`、`hasRole` 等规则，因此：

- **RestAccessDeniedHandler** 在当前配置下**不会被触发**（URL 级只校验「是否登录」，无权限校验）
- 权限由 Controller 上的 **@PreAuthorize** 控制，失败时走 **GlobalExceptionHandler**
- 若 RestAccessDeniedHandler 未生效，属预期行为

**RestAccessDeniedHandler 触发示例：**

若在 SecurityConfig 中增加 URL 级权限规则，例如：

```java
.antMatchers(GET, "/admin/user/list").hasAuthority("ums:admin:list")
.antMatchers(POST, "/admin/user").hasAuthority("ums:admin:add")
```

则会出现以下流程：

1. 用户 A 已登录，但仅有 `ums:role:list`，无 `ums:admin:list`
2. 请求 `GET /admin/user/list`
3. 请求进入 `FilterSecurityInterceptor`
4. 规则要求 `ums:admin:list`，用户无此权限 → 抛出 `AccessDeniedException`
5. `ExceptionTranslationFilter` 捕获 → 调用 **RestAccessDeniedHandler#handle**
6. 返回 403 JSON，请求**不会进入 Controller**

此时 URL 级鉴权在过滤链中已完成，RestAccessDeniedHandler 才会被调用。

## 五、过滤器链详解

### 5.1 过滤器执行顺序

Spring Security 默认有一系列过滤器，本项目在 `UsernamePasswordAuthenticationFilter` **之前**插入了自定义的 `JwtAuthenticationTokenFilter`：

```
请求 → ... → JwtAuthenticationTokenFilter → UsernamePasswordAuthenticationFilter → ... → Controller
```

### 5.2 JwtAuthenticationTokenFilter 详解

路径：`fireworks-admin/src/main/java/com/fireworks/admin/security/JwtAuthenticationTokenFilter.java`

继承 `OncePerRequestFilter`，保证每次请求最多执行一次。

**核心逻辑：**

```
1. 读取请求头 Authorization
2. 若无或非 "Bearer xxx" 格式 → 直接放行（不做任何事）
3. 若有 → 截取 token，用 JwtTokenUtil 解析出 username
4. 从 Redis 取 AdminUserDetails（key: user:info:{username}）
5. 若 Redis 无数据 → 会话过期或已被踢下线 → 放行（后续授权判断失败 → 401）
6. 若有数据 → validateToken 校验（username 一致）
7. 校验通过 → 构造 Authentication，写入 SecurityContextHolder
8. 刷新 Redis TTL（30 分钟滑动窗口）
9. 放行
```

**设计要点：**

- **不查 DB**：用户信息从 Redis 读取，减少 DB 压力
- **会话由 Redis 控制**：登录写入、修改密码/禁用时删除，实现「踢下线」
- **滑动窗口**：每次有效请求刷新 TTL，用户持续活跃则不掉线

### 5.3 为什么登录接口不需要 Token 能通过？

1. `/admin/login` 被 `permitAll()` 放行，**不要求已认证**
2. `JwtAuthenticationTokenFilter` 发现无 Token 时直接 `doFilter` 放行
3. 请求到达 `FilterSecurityInterceptor` 时，匹配到 `permitAll`，直接放行到 Controller
4. Controller 正常执行登录逻辑

## 六、关键组件说明

### 6.1 AdminUserDetailsService

| 职责 | 说明 |
|------|------|
| 接口 | `UserDetailsService` |

| 作用 | 从 DB 加载管理员及权限、角色，封装为 `AdminUserDetails` |
| 调用时机 | 登录时由 `UmsAdminServiceImpl.login()` 调用 |

### 6.2 AdminUserDetails

| 职责 | 说明 |
|------|------|
| 接口 | `UserDetails` |
| 内容 | UmsAdmin、权限列表、角色列表 |
| 序列化 | 可被 Jackson 序列化存入 Redis |

### 6.3 JwtTokenUtil

| 方法 | 作用 |
|------|------|
| `generateToken(UserDetails)` | 生成 JWT，subject 为 username |
| `getUsernameFromToken(String)` | 从 token 解析 username |
| `validateToken(token, userDetails)` | 校验 subject 与 userDetails 用户名一致 |

### 6.4 Redis 会话

| Key | 格式 | 过期 | 说明 |
|-----|------|------|------|
| 用户会话 | `user:info:{username}` | 30 分钟 | 存 AdminUserDetails，登录写入、修改密码/禁用删除 |

## 八、常见问题速查

| 问题 | 可能原因 | 排查方向 |
|------|----------|----------|
| 登录后访问接口返回 401 | Token 未携带或 Redis 无会话 | 检查 Authorization 头、Redis 中 key 是否存在 |
| 修改密码后旧 Token 仍可用 | 未清除 Redis | 确认 `updateProfile`/`updateAdmin` 中调用了 `RedisUtil.delete` |
| 401 返回 HTML 而非 JSON | 未注册 RestAuthenticationEntryPoint | 检查 SecurityConfig 的 exceptionHandling |
| 有 Token 但返回 403 | 权限不足 | 检查 @PreAuthorize 与用户权限列表 |
| 登录接口返回 401 | 被误拦截 | 确认 /admin/login 在 permitAll 中 |
| RestAccessDeniedHandler 从不触发 | 当前只用 @PreAuthorize，无 URL 级 hasAuthority 规则 | 见 4.5 节；方法级 403 由 GlobalExceptionHandler 处理 |
