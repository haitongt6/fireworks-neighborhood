## 4. 必补知识点（按优先级排序）

下面这些知识点，是你现在最值得补齐的部分。按优先级从高到低排列。

---

## 4.1 第一优先级：JVM 内存结构（排查导向版）

### 你必须搞清楚的内容

#### 1）Heap 是什么

- 主要存放 Java 对象
- Heap 泄漏、`Java heap space` OOM 主要发生在这里

#### 2）新生代 / 老年代 是什么

- 新生代：新创建的对象通常先在这里
- 老年代：活得久的对象、晋升后的对象通常在这里

#### 3）Metaspace 是什么

- 存放类元数据，不是普通对象
- `Metaspace OOM` 和 `Heap OOM` 不是一回事

#### 4）线程栈是什么

- 存放方法调用栈帧
- CPU 排查中看到的 thread dump，本质上是在看线程栈

### 你学到什么程度算够

只要你能做到下面这句能顺口讲出来，就算过关：

> Heap 放对象，Metaspace 放类信息，线程栈放方法调用现场；CPU 问题优先看线程栈，内存泄漏优先看 Heap，对象为什么活着最终要靠引用链来定案。

---

## 4.2 第二优先级：GC 基础概念

### 你必须搞清楚的内容

#### 1）Young GC 是什么

- 主要回收新生代
- 通常较快、较常见

#### 2）Full GC 是什么

- 比 Young GC 更重
- 通常值得重点关注

#### 3）什么情况下 Full GC 值得警惕

- `FGC` 持续增长
- Full GC 后老年代占用仍然很高
- GC 次数越来越多，但内存降不下来

### 你学到什么程度算够

你至少要能讲出：

> `jstat -gcutil` 里，`YGC` 和 `FGC` 看的是累计次数；判断压力不能只看绝对值，要看在观察窗口内是否持续增长，尤其是 Full GC 后老年代是否还降不下来。

---

## 4.3 第三优先级：GC Root 与引用链

### 你必须搞清楚的内容

#### 1）什么是 GC Root

- 可以理解成：GC 判断对象是否存活时的“根”
- 只要对象还能沿引用链追溯到 GC Root，它就不会被回收

#### 2）为什么“GC 没回收”不等于“GC 不工作”

- 更常见的原因是：对象还被强引用着
- 比如静态字段、集合、单例、ThreadLocal 等

#### 3）Heap 泄漏为什么最终要靠 MAT 定案

- 因为 MAT 可以告诉你对象为什么活着
- 关键是 `Path To GC Roots`

### 你学到什么程度算够

你至少要能讲出：

> 内存泄漏的关键不是“对象很多”，而是“对象本该死但还活着”，而判断它为什么还活着，最终要看它到 GC Root 的强引用链。

---

## 4.4 第四优先级：CPU 排查的核心逻辑

### 你必须搞清楚的内容

#### 1）`RUNNABLE` 的真实含义

- 表示线程处于可运行 / 正在运行状态
- 不等于一定高 CPU

#### 2）为什么单次 `Thread.print` 不足以定案

- 它只是瞬时快照
- 有可能线程只是“碰巧正在运行”

#### 3）为什么 Arthas `cpuUsage` 更直接

- 它是基于采样窗口内线程 CPU 时间增量估算的
- 更接近“谁在这段时间真的最忙”

### 你学到什么程度算够

你至少要能讲出：

> `Thread.print` 更像线程现场照片，而 Arthas `thread -n 5` 更像给线程按 CPU 使用率做排序；如果没有 Arthas，就要通过多次 `Thread.print` 提高判断准确性。

---

## 4.5 第五优先级：MAT 的 4 个核心视图

### 你必须搞清楚的内容

#### 1）Histogram

看：

- 有哪些对象
- 数量多不多
- 占用大不大

#### 2）Dominator Tree

看：

- 谁在“保留”这些对象
- 谁把内存卡住了

#### 3）Path To GC Roots

看：

- 对象为什么活着
- 它是被哪条强引用链挂住的

#### 4）Leak Suspects

看：

- MAT 自动生成的嫌疑报告
- 适合快速锁定第一嫌疑对象

### 你学到什么程度算够

你至少要能讲出：

> `Leak Suspects` 适合快速圈嫌疑人，`Dominator Tree` 用来看谁在保留内存，`Path To GC Roots` 才适合最终确认为什么对象不能回收。

## 5. 面试里最容易混淆的知识点（重点警惕）

下面这些点，是面试中非常容易被追问、也非常容易说错的地方。

### 5.1 `jstat` 不是抓线程栈的

错误理解：

- `jstat` 可以抓线程快照
- `jstat` 可以做 heap dump

正确理解：

- `jstat` 主要看 JVM 统计指标
- 线程栈看 `jcmd Thread.print` 或 `jstack`
- heap dump 用 `jcmd GC.heap_dump`

---

### 5.2 `RUNNABLE` 不等于热点线程

错误理解：

- 看到 `RUNNABLE` 就是它打高 CPU

正确理解：

- `RUNNABLE` 只是候选条件
- 还要结合线程类型、调用栈、连续采样、Arthas `cpuUsage`

---

### 5.3 `jstat` 不能直接证明内存泄漏

错误理解：

- `O` 高了就是泄漏
- `FGC` 多了就是泄漏

正确理解：

- `jstat` 只能提示“压力和趋势”
- 真正定性泄漏，还是要看 dump 和引用链

---

### 5.4 Heap OOM 和 Metaspace OOM 不能混

错误理解：

- OOM 都是一种问题

正确理解：

- `Java heap space`：对象太多或无法回收
- `Metaspace`：类元数据太多或 Metaspace 太小

## 7. 建议你练成的“标准回答模板”

为了防止面试时说乱，建议你对以下几个问题形成固定模板。

### 7.1 CPU 高排查模板

```text
先确认目标 Java 进程 PID；
再抓线程快照，看哪些线程处于 RUNNABLE；
如果有 Arthas，优先用 thread -n 5 看热点线程；
结合线程栈确认是否停在业务代码；
必要时结合 jstat -gcutil 判断 GC 是否也在参与；
最终定位是业务线程热点还是 GC 压力。
```

---

### 7.2 Heap 泄漏排查模板

```text
先看业务现象或接口状态，确认对象在持续累积；
再用 jstat -gcutil 看堆和 GC 趋势，但它只能提示压力，不能直接定案；
在 OOM 前用 jcmd GC.heap_dump 抓现场；
用 MAT 先看 Leak Suspects，再结合 Dominator Tree 和 Path To GC Roots；
如果发现对象通过静态字段/集合一直连到 GC Root，就能确认是泄漏；
最后通过 reset 或清理动作验证问题是否解除。
```

---

### 7.3 Metaspace OOM 模板

```text
Metaspace OOM 不是普通对象太多，而是类元数据占用过高；
常见原因包括动态代理、脚本生成类、类加载器泄漏、Metaspace 上限过小；
排查时重点看类加载情况、Metaspace 使用情况和 classloader；
不能把它和 Heap OOM 混为一谈。
```
