# JVM 内存结构（排查导向版）

> 目标：不是从学术角度死记 JVM 所有术语，而是从**线上排查**角度，把 JVM 内存结构真正讲清楚。
>
> 适用人群：已经做过 `cpu-burn`、`heap-leak`、`metaspace-leak` 演练，但对 Heap、栈、Metaspace、新生代、老年代、Young GC、Full GC 等概念还没有形成清晰脑图的同学。
>
> 阅读目标：看完后，能把 JVM 内存结构和你已经做过的三类演练、常见工具输出、常见 OOM 类型真正对上号。

---

## 1. 为什么很多人看完 JVM 内存结构文档还是“脑子里很散”

很多 JVM 文档是按“知识分类”写的，会依次介绍：

- 程序计数器
- 虚拟机栈
- 本地方法栈
- 堆
- 方法区
- 常量池
- Metaspace

这些内容本身没问题，但如果你当前是从**线上排查**角度学习，就很容易出现一个问题：

> 每个词都看过，但不知道排查时它们分别对应什么现象，也不知道自己到底该重点学哪几个。

所以这份文档不按“学术顺序”讲，而是按下面这个问题来讲：

> **当线上出现 CPU 高、Heap 泄漏、Metaspace OOM 时，到底是哪一块出了问题？我该如何在脑子里快速定位到 JVM 的哪一部分？**

---

## 2. 先建立总图：JVM 内存结构分成两大类

你可以先把 JVM 内存结构分成两类：

### 2.1 线程私有

每个线程自己单独拥有一份，互不共享：

- 程序计数器（PC Register）
- Java 虚拟机栈（JVM Stack）
- 本地方法栈（Native Method Stack）

### 2.2 线程共享

整个 JVM 进程内多个线程共同使用：

- Heap（堆）
- 方法区 / Metaspace（JDK 8 重点看 Metaspace）
- 运行时常量池（通常归到方法区理解）

### 2.3 排查时最常用的是哪几块

如果你从排查角度看，最常打交道的是：

- **线程栈**：CPU 问题看这里
- **Heap**：对象泄漏、堆 OOM 看这里
- **Metaspace**：类元数据问题、Metaspace OOM 看这里

所以现阶段你不需要把所有区域学得一样深。最重要的是先把这三块真正搞明白。

## 3. 从排查角度理解 JVM 内存结构

下面按“线上排查最有用”的顺序讲。

---

## 4. 线程栈（Java 虚拟机栈）：CPU 排查的主战场

### 4.1 它是什么

当一个线程执行 Java 方法时，会在虚拟机栈中创建一层层栈帧。

栈帧里主要会保存：

- 方法参数
- 局部变量
- 返回地址
- 一些中间运算状态

你可以把它理解成：

> **线程当前正在执行哪些方法、从哪一层调到哪一层的“现场记录本”。**

### 4.2 为什么它对 CPU 排查很重要

因为当 CPU 飙高时，你真正想知道的是：

- 哪个线程最忙
- 它在执行哪段代码
- 调用链走到了哪里

而这些信息，都是通过线程栈看到的。

### 4.3 和你做过的演练怎么对应

你前面做过：

```text
/api/internal/jvm-practice/cpu-burn?ms=120000
```

抓线程栈时看到：

```text
http-nio-8080-exec-5
RUNNABLE
at com.fireworks.api.controller.ApiJvmPracticeController.cpuBurn(...)
```

这本质上就是在看：

> **某个线程的虚拟机栈当前停在 `cpuBurn()` 这个方法上。**

### 4.4 你需要记住的排查结论

- CPU 高，优先想线程栈
- `Thread.print` / Arthas `thread -n 5` 都是在看线程栈相关现场
- 线程栈不是用来看内存泄漏的，而是用来看“谁在执行什么代码”的

---

## 5. Heap（堆）：对象内存问题的主战场

### 5.1 它是什么

Heap 是：

> **绝大多数 Java 对象分配和存活的主要区域。**

常见对象基本都在 Heap：

- `new byte[]`
- `new ArrayList<>()`
- `new HashMap<>()`
- 业务对象、DTO、VO、缓存对象

### 5.2 为什么它对内存泄漏排查最重要

因为所谓“Java 内存泄漏”，大多数场景里其实指的是：

> **Heap 中的对象本来应该被回收，但因为还被引用着，所以一直活着，最终把堆慢慢撑满。**

### 5.3 和你做过的演练怎么对应

你做的：

```text
/api/internal/jvm-practice/heap-leak/leak?memberId=1&kb=8192
```

接口每次都会：

- 创建一个 `byte[]`
- 把它放进静态 `ConcurrentHashMap<Long, List<byte[]>>`

这些 `byte[]` 都是在 Heap 里。

MAT 里你看到：

```text
ApiJvmPracticeController
 -> ConcurrentHashMap
 -> ArrayList
 -> Object[]
 -> byte[]
```

这就是典型的 Heap 泄漏证据链。

### 5.4 你需要记住的排查结论

- Heap 里放对象
- Heap 泄漏最终要看对象为什么一直活着
- `jstat -gcutil` 只能看堆和 GC 趋势
- 真正确认 Heap 泄漏，要靠 `heap dump + MAT + 引用链`

---

## 6. Heap 里的新生代和老年代：为什么 `jstat` 里会看到 `E` 和 `O`

Heap 不是一个没有结构的大池子。JVM 通常会把 Heap 分代。

### 6.1 新生代（Young Generation）

主要放：

> **刚创建出来、通常寿命不长的对象**

新生代内部常见还会再分：

- Eden
- Survivor 0（S0）
- Survivor 1（S1）

#### 最通俗理解

- 新对象先放 Eden
- 经过 Young GC 后，如果对象还活着，会在 Survivor 区之间转移
- 活得够久的对象，可能会晋升到老年代

### 6.2 老年代（Old Generation）

主要放：

> **存活时间较长、经历过多次 GC 仍然活着的对象**

### 6.3 为什么这对排查重要

因为很多 Heap 泄漏对象最终都会进入老年代。

所以你在 `jstat -gcutil` 里看到：

- `E`：大致代表 Eden 使用率
- `O`：大致代表 Old（老年代）使用率

当你做 `heap-leak` 演练时，如果对象一直不释放，通常会慢慢表现为：

- `O` 越来越高
- Full GC 之后 `O` 仍然降不下来

这时你就要高度怀疑：

> 这些对象可能已经变成长生命周期对象，被老年代持续持有。

### 6.4 你该学到什么程度

现阶段不用深挖对象晋升的所有细节，只要先建立下面这个认知：

> **新生代放新对象，老年代放活得久的对象；老年代长期高位且 Full GC 后不下降，是排查 Heap 泄漏时的重要信号。**

---

## 7. Young GC 和 Full GC：你在 `jstat -gcutil` 里真正该看什么

### 7.1 Young GC 是什么

Young GC 主要回收新生代。

特点通常是：

- 比较频繁
- 耗时相对较低
- 不一定说明系统有问题

### 7.2 Full GC 是什么

Full GC 通常比 Young GC 更重，常见会涉及老年代的回收。

特点通常是：

- 更慢
- 更值得关注
- 如果频繁发生，往往是一个警讯

### 7.3 `jstat -gcutil` 中相关列

你最常用的是：

- `YGC`：Young GC 次数
- `YGCT`：Young GC 总耗时
- `FGC`：Full GC 次数
- `FGCT`：Full GC 总耗时
- `GCT`：GC 总耗时

### 7.4 应该怎么理解这些列

不要只看某一行的绝对值，而要看：

> **在你观察的这段时间里，它们有没有继续增长，增长速度如何。**

#### 如果你看到：

- `YGC` 有增长，但系统整体还稳

说明：

- 对象创建活跃，Young GC 正常发生
- 不一定异常

#### 如果你看到：

- `FGC` 开始持续增长
- Full GC 后 `O` 仍然很高

说明：

- 老年代压力越来越大
- 值得重点怀疑 Heap 泄漏或长生命周期对象过多

### 7.5 你现在最该建立的认知

> `jstat -gcutil` 是看 JVM 内存压力和 GC 趋势，不是直接证明泄漏的工具。

---

## 8. Metaspace：类元数据问题的主战场

### 8.1 它是什么

在 JDK 8 里，Metaspace 用来存放：

> **类的元数据**

例如：

- 类定义信息
- 方法、字段元信息
- 某些常量池相关信息
- 动态生成的类信息（代理类、脚本类等）

### 8.2 最通俗理解

如果说 Heap 放的是“对象”，
那 Metaspace 放的更像是“类的说明书”。

### 8.3 和你做过的演练怎么对应

你前面碰到过：

```text
java.lang.OutOfMemoryError: Metaspace
```

那不是普通对象太多，而是：

- 类元数据太多
- Metaspace 上限太小
- 或动态生成类太多

`metaspace-leak` 演练本质就是在制造这类压力。

### 8.4 为什么 Heap OOM 和 Metaspace OOM 不能混

#### Heap OOM

```text
OutOfMemoryError: Java heap space
```

通常意味着：

- Heap 中对象太多
- 对象没被回收
- 大集合 / 缓存 / 泄漏 / 堆太小

#### Metaspace OOM

```text
OutOfMemoryError: Metaspace
```

通常意味着：

- 类元数据压力过大
- 动态代理 / 脚本 / 类加载器问题
- Metaspace 上限太小

### 8.5 你现在最该建立的认知

> Heap OOM 是对象问题，Metaspace OOM 是类信息问题。它们都叫 OOM，但根因和排查方向完全不同。

---

## 9. 程序计数器与本地方法栈：知道存在即可

### 9.1 程序计数器（PC Register）

它可以理解成：

> 当前线程下一条要执行的字节码位置记录器

作用类似“执行位置的小书签”。

### 9.2 本地方法栈（Native Method Stack）

主要服务于 native 方法的调用。

### 9.3 当前阶段应该怎么学

现阶段你知道它们存在即可。对于你现在的排查目标：

- CPU 高：重点还是线程栈
- Heap 泄漏：重点还是堆
- Metaspace：重点还是类元数据区

不需要在 PC Register 和 Native Method Stack 上投入太多精力。

## 10. 从排查角度重新理解三类问题

下面把你已经做过的三个演练，重新和 JVM 内存结构做一次映射。

---

## 10.1 `cpu-burn`

### 对应 JVM 视角

- 线程栈
- 线程状态
- CPU 时间消耗

### 为什么不是 Heap 问题

因为它不是对象大量堆积，而是：

- 某个线程一直在执行业务方法
- 导致 CPU 被持续占用

### 你用了什么工具

- `jcmd Thread.print`
- Arthas `thread -n 5`

### 你在看什么

- 哪个线程是热点线程
- 栈顶是不是业务方法
- 是否持续停在同一位置

---

## 10.2 `heap-leak`

### 对应 JVM 视角

- Heap
- 新生代 / 老年代
- 引用链
- GC 是否能回收

### 为什么不是栈问题

因为问题核心不是“哪个线程在执行”，而是：

- 哪些对象活着
- 为什么活着
- 为什么 GC 回收不了

### 你用了什么工具

- `jstat -gcutil`
- `jcmd GC.heap_dump`
- MAT

### 你在看什么

- `O` 是否升高
- `FGC` 是否增长
- `byte[]` 是否被静态容器持有
- 对象是否有通向 GC Root 的强引用链

---

## 10.3 `metaspace-leak`

### 对应 JVM 视角

- Metaspace
- 类加载
- 动态生成类
- 类元数据占用

### 为什么不是普通 Heap 泄漏

因为它不是普通对象太多，而是：

- 类信息增长
- 类加载器 / 代理 / 脚本类问题

### 你常用的观察方向

- `jstat -class`
- `classloader -l`
- `VM.native_memory summary`

---

## 11. 你看工具输出时，脑子里应该自动浮现什么

这一节非常重要，它能帮助你真正做到“心中有数”。

### 11.1 看到 `Thread.print`

你脑子里要想：

> 我现在在看线程栈，目标是找哪个线程在执行哪段代码。

### 11.2 看到 `jstat -gcutil`

你脑子里要想：

> 我现在在看 Heap 和 GC 的趋势，目标是判断内存压力是否在加重，但还不能直接定案泄漏。

### 11.3 看到 MAT 的 `Dominator Tree`

你脑子里要想：

> 我现在在看谁把内存“卡住”了，谁在保留这些对象。

### 11.4 看到 MAT 的 `Path To GC Roots`

你脑子里要想：

> 我现在在看对象为什么不能被 GC 回收，它到底通过哪条强引用链连到了 GC Root。

### 11.5 看到 `OutOfMemoryError: Metaspace`

你脑子里要想：

> 这不是普通堆对象爆了，而是类元数据区出问题了。

## 12. 你当前最应该建立的“最小脑图”

现阶段不需要把 JVM 学成底层专家，但你至少应该在脑子里有这张图：

```text
JVM 内存结构
├─ 线程私有
│  ├─ 程序计数器：记录线程执行位置
│  ├─ 虚拟机栈：方法调用栈，CPU 排查重点
│  └─ 本地方法栈：native 方法相关
│
└─ 线程共享
   ├─ Heap：放对象，内存泄漏 / Heap OOM 排查重点
   │  ├─ 新生代：新对象
   │  └─ 老年代：活得久的对象
   │
   └─ Metaspace：放类元数据，类加载 / Metaspace OOM 排查重点
```

如果你能把这张图真正记住，很多概念就不会散。

## 13. 你应该学到哪一步比较合适

很多人学 JVM 容易走两个极端：

### 极端 1：只会命令，不懂结构

问题是：

- 会执行工具，但不知道看到的现象说明什么

### 极端 2：一上来硬啃所有 JVM 底层原理

问题是：

- 内容太多，容易崩
- 对当前排查帮助不大

### 对你当前最合适的学习目标

学到下面这个程度就很好：

> **看到 CPU、Heap、Metaspace 相关现场后，能判断大方向，知道先抓什么，知道最终靠什么定案。**

也就是说，你现在最适合的是：

- 学会区分 Heap / Metaspace / 栈
- 学会区分 Young GC / Full GC
- 学会区分“压力指标”和“根因证据”
- 学会把工具和 JVM 区域一一对应起来

这已经足够支持大多数工作场景和面试基础题。

## 14. 建议你的学习顺序（不要乱）

### 第一阶段：先补骨架

重点学：

- Heap
- 新生代 / 老年代
- Metaspace
- 线程栈
- Young GC / Full GC
- GC Root

目标：

- 看工具输出时不慌
- 脑子里能对应到 JVM 哪一块

### 第二阶段：和演练绑定着学

- `cpu-burn` 对应线程栈与热点线程
- `heap-leak` 对应 Heap 与引用链
- `metaspace-leak` 对应 Metaspace 与类加载

目标：

- 每学一个概念，都知道它在现场里长什么样

### 第三阶段：以后再逐步补底层

例如：

- 更深入的 GC 算法
- 类加载细节
- 字节码
- JIT / 锁优化 / JMM

这部分以后再补，不是你当前最优先的。

## 16. 一句话总结

如果你只记一句话，请记住：

> **CPU 问题优先看线程栈，Heap 泄漏优先看对象和引用链，Metaspace 问题优先看类元数据与类加载；看到现场时，先判断它属于 JVM 的哪一块，再决定用什么工具。**
